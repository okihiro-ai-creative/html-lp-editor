# HTML LP Editor — HTML LP編集プロトコル

---

## ⚡ このスキルの使い方

スキルを呼び出されたら、以下の**3フェーズを必ず順番に実行**せよ。

### 【必須 1】PC / SP 改行分離CSS

**なぜ最初にやるのか：**
PCとSPの改行を同じ `<br>` で管理すると、PCの改行を調整するたびにSPの改行も崩れ、
全SP箇所を再調整する羽目になる。先に分離しておくことで**後工数をゼロにする**。

**本スキルの鉄則：**
- **無印の `<br>` は絶対に使用禁止。** 改行を入れる場合は必ず `<br class="br-pc">` または `<br class="br-sp">` のいずれかを使うこと。無印 `<br>` はPC/SP両方で改行されてしまい、片方だけ制御できなくなる。
- **PCの `<br class="br-pc">` だけを調整して改行を決める**
- **SPは自然折り返しに任せる（手動の `<br class="br-sp">` は原則使わない）**
- SPで手動改行を入れると、端末幅によって数文字だけはみ出す「泣き別れ」事故が起きる

```css
/* PC/SP 改行分離 — 必ずstyle内の先頭付近に記述 */
@media (max-width: 640px) {
  .br-pc { display: none; }
}
@media (min-width: 641px) {
  .br-sp { display: none; }
}
```

**HTML側の使い方：**
```html
<!-- PCでだけ改行したい箇所 -->
髪が整うと、<br class="br-pc">
朝の余白が変わる。

<!-- SPでだけ改行したい箇所（原則使わない。使う場合は短い見出しのみ） -->
タイトル<br class="br-sp">テキスト
```

**⚠️ 長文・説明文への適用禁止：**
`br-sp` を長文の説明文に使うと、端末幅によって残りの数文字だけが孤立する。
長文はブラウザの自然な折り返しに完全に任せること。

---

### 【必須 2】はみ出し完全封殺（2層構造）

**設計思想：**
`overflow-x: hidden` は「はみ出したものを隠す」だけで、**はみ出し自体を防がない**。
隠されたコンテンツは見えなくなるだけで、レイアウト上は親を突き破ったまま。
つまり `overflow-x: hidden` だけでは**泣き別れ・文字切れ・横スクロールの根本解決にならない**。

**必ず2層構造で実装せよ：**
```
Layer 1（封じ込め）→ そもそもはみ出しを発生させない
Layer 2（最後の砦）→ 万が一のはみ出しを隠す
```

---

#### Layer 1：封じ込め — はみ出しを「作らせない」CSS

**これが本体。全プロジェクトの `<style>` 冒頭に必ず入れよ：**

```css
/* ===== Layer 1: 封じ込め ===== */

/* 全要素を親の幅内に収める */
*, *::before, *::after {
  box-sizing: border-box;
  max-width: 100%;
}

/* 全テキスト要素：親をはみ出す長い単語を強制折り返し */
h1, h2, h3, h4, h5, h6,
p, li, summary, blockquote, a, span, td, th {
  overflow-wrap: anywhere;
  word-break: break-word;
}

/* gridの子要素：min-width: auto がデフォルトで親をはみ出す */
/* grid を使うすべての親の直下に適用 */
[style*="grid"], .grid, [class*="grid"] {
  min-width: 0;
}
[style*="grid"] > *, .grid > *, [class*="grid"] > * {
  min-width: 0;
}

/* flexの子要素：同様に min-width: 0 が必要 */
[style*="flex"] > *, .flex > *, [class*="flex"] > * {
  min-width: 0;
}

/* imgとvideoは絶対にはみ出させない */
img, video, iframe, canvas, svg {
  max-width: 100%;
  height: auto;
}
```

---

#### Layer 2：最後の砦 — 万が一のはみ出しを隠す

**Layer 1 で防ぎきれない特殊ケース（外部埋め込み等）への保険：**

```css
/* ===== Layer 2: 最後の砦 ===== */
html,
body {
  max-width: 100%;
  overflow-x: clip;
}
main {
  overflow-x: clip;
}
```

**スマホ用メディアクエリにも必ず記述：**
```css
@media (max-width: 640px) {
  html,
  body,
  main {
    max-width: 100%;
    overflow-x: clip;
  }
}
```

**⚠️ なぜ `hidden` ではなく `clip` を使うか（重要）：**
`overflow-x: hidden` は内部的に `overflow-y` も `auto` に昇格させ、**新しいスクロールコンテナを生成**する。これにより `position: sticky` を中に置いた要素（追従ヘッダー等）は viewport ではなくその新コンテナにスティックしようとして、**機能しなくなる**。
`overflow-x: clip` は視覚的には `hidden` と同じ効果だがスクロールコンテナを生成しないので、sticky が正常動作する。

| プロパティ | 視覚効果 | スクロールコンテナ生成 | sticky への影響 |
|---|---|---|---|
| `overflow-x: hidden` | はみ出し非表示 | **する**（副作用大） | sticky 壊れる |
| `overflow-x: clip` | はみ出し非表示 | しない | sticky 動作する |

ブラウザ対応：Chrome 90+, Firefox 91+, Safari 16+（モダンブラウザのシェア99%+）

---

#### ⛔ はみ出しを「作る」CSSプロパティ禁止リスト

以下のプロパティは**はみ出しの直接原因**になる。使用を固く禁ずる：

| 禁止プロパティ | なぜ危険か | 代替手段 |
|---|---|---|
| `white-space: nowrap` | テキストが折り返せず親をはみ出す | 使わない。長い見出しは `clamp()` で縮小 |
| `width: 100vw` | スクロールバー幅を含み親より広くなる | `width: 100%` を使う |
| `min-width: ○○px`（大きい値） | 画面幅より大きくなり得る | `min-width: min(○○px, 100%)` を使う |
| `flex-shrink: 0`（長いテキストに） | テキストが縮小されず親をはみ出す | テキスト要素には使わない |
| `position: absolute` + 大きな `left/right` | 画面外に配置されはみ出す | `overflow: hidden` の親で囲む |

### 【知識 1】WordPress クラス名衝突パターン

WordPressテーマは以下のありふれたクラス名を自前で持っていることが多い。
LP側でこれらを使うと、テーマのCSSが誤爆して予期しないデザイン崩れが起きる。

**衝突しやすいクラス名一覧（使用禁止）：**

| 危険なクラス名 | 代替案 |
|--------------|--------|
| `.header` | `.site-header` / `.lp-header` |
| `.footer` | `.site-footer` / `.lp-footer` |
| `.nav` | `.lp-nav` / `.main-nav` |
| `.wrap` | `.lp-wrap` / `.inner` |
| `.hero` | `.lp-hero`（テーマ依存・要確認） |
| `.container` | `.lp-container` |
| `.section` | `.lp-section` |

**今回の実例（特定サイトでの事例）：**
`.header { backdrop-filter: blur(18px); }` がWordPress側の巨大な `<header>` に誤爆し、
画面全体がすりガラス状態（真っ白）になった。→ `.site-header` にリネームして解決。

### 【知識 2】レスポンシブタイポグラフィ

固定px指定は端末によって大きすぎたり小さすぎたりする。
`clamp()` を使って「最小・推奨・最大」の3点で指定せよ。

```css
/* Hero h1の例 */
h1 { font-size: clamp(28px, 7.5vw, 76px); }

/* セクションh2の例 */
h2 { font-size: clamp(24px, 5vw, 58px); }

/* CTAのh2（スマホ優先）*/
.cta-inner h2 { font-size: clamp(28px, 8vw, 76px); }
```

### 【知識 3】カードバランスの考え方

**本スキルの優先順位：**
1. 改行位置とマージンによる余白の「呼吸」
2. カード同士の高さ・大きさの統一感
3. 情報密度のバランス（詰め込みすぎない・空きすぎない）

**grid カード設計の基本：**
```css
/* カードが自動で最適列数になるパターン */
.card-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(min(100%, 280px), 1fr));
  gap: 16px;
}
```

**固定カラム数は避ける（スマホで50%ずつに潰れる）：**
```css
/* NG */
grid-template-columns: 1fr 1fr;

/* OK */
grid-template-columns: repeat(auto-fit, minmax(min(100%, 280px), 1fr));
```

### 【知識 4】flex レイアウトの必須ルール

/* 基本：折り返しを許可 */

### 【知識 5】`<style>` タグの位置

`<style>` は必ず `<head>` の中に書くこと。
`<body>` 直下に書くと、WordPress等の環境でパースエラーが起きてページが真っ白になる原因になる。

```html
<!-- 正しい構造 -->
<html>
  <head>
    <meta charset="utf-8">
    <style>
      /* ここに全CSSを書く */
    </style>
  </head>
  <body>
    <!-- コンテンツ -->
  </body>
</html>
```

### 【知識 7】日本語見出しの折返し最適化

**`text-wrap: balance` は日本語に不向き — 絶対に使うな。**

`text-wrap: balance` は英語ベースの折返しアルゴリズムで「単語境界（半角スペース）」を前提に上下の文字数を均等化しようとする。日本語、特に**カタカナ連続**には単語境界が無いため、アルゴリズムが「とにかく文字数を均等」にしようと**カタカナの途中で改行**する事故が起きる。

**事故例（2026-05-15 LP）：**
```
無料診断＆オ
ンライン面談     ← カタカナ "オ" / "ン" で分断
```

**正しい設定：**
```css
/* 日本語ページの見出し折返しはこれだけで十分 */
h1, h2, h3 {
  text-wrap: pretty;
}
p {
  text-wrap: pretty;
}
```

`text-wrap: pretty` はバランス強制をせず、widow/orphan（1文字だけ取り残された行）の回避のみを行うので日本語と相性が良い。

**追加禁止：`word-break: keep-all` も日本語に副作用あり。**
「、」「。」の前で改行禁止のルールが効きすぎて、句読点が**単独で取り残される**事故が起きる。Phase 1 Layer 1 で既にグローバルに `overflow-wrap: anywhere; word-break: break-word;` が入っているので、それに任せれば良い。`keep-all` を上書きしない。

### 【知識 8】"×" 文字（U+00D7）はベースラインずれの罠

閉じるボタン・モーダルクローズ等で `×` 文字を直接使うと、**多くのフォントで縦中央に表示されない**。U+00D7（MULTIPLICATION SIGN）はベースラインが下寄りに設計されており、`display: flex; align-items: center` でも円の中で下にずれる。

**事故例（2026-05-15 LP）：**
```html
<!-- NG：×が円の中で下に偏る -->
<button class="zoom-close" aria-label="閉じる">×</button>
```

**正しい実装：SVG に置換**
```html
<button class="zoom-close" aria-label="閉じる">
  <svg viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2.4" stroke-linecap="round" aria-hidden="true">
    <line x1="6" y1="6" x2="18" y2="18"/>
    <line x1="18" y1="6" x2="6" y2="18"/>
  </svg>
</button>
```

```css
.zoom-close svg { width: 18px; height: 18px; }
```

SVGは viewBox 内で完全に対称なので、フォント依存のベースラインずれが起きない。

「×」だけでなく「＋」「→」等の記号も同様の事故が起きる。**装飾記号は基本SVG化**が安全。

### 【知識 9】動画 `poster` 属性の運用注意

`<video poster="...">` は動画再生開始までの初期表示画像。**この画像が 404 だと初期表示が空白**になり、ヒーロー領域がスカスカに見える致命的事故になる。

**事故例（2026-05-15 LP）：**
動画を `-5.mp4` から `-6.mp4` に差し替えた際、poster 画像（`-3x1.png`）が同期更新されておらず 404。ヒーロー領域が一瞬白く抜ける状態に。

**運用ルール：**
1. 動画差し替え時は **poster 画像も同時に差し替える**
2. または **poster 属性を外し**、`.hero-video` の `background` で代替（地味だが安全）
3. 動画再生開始の検知は `requestVideoFrameCallback` か `playing` イベントを使う（`canplay` / `loadeddata` は早すぎる）

**動画再生開始まで待つローダー解除の正解：**
```js
function waitForHeroVideo() {
  var video = document.getElementById('heroVideo');
  if (!video) { hide(); return; }
  if (typeof video.requestVideoFrameCallback === 'function') {
    // 実フレーム描画を待つ（最強）
    video.requestVideoFrameCallback(hide);
  } else {
    // フォールバック：再生開始イベント
    video.addEventListener('playing', hide, { once: true });
  }
  video.addEventListener('error', hide, { once: true }); // 失敗時もローダー残さない
}
// セーフティ：最大15秒
setTimeout(hide, 15000);
```

**使ってはいけないイベント：** `canplay` / `loadeddata` / `stalled` は「データが先読みされた」だけで実フレームは描画されていない。これらで `hide()` するとローダーが**動画表示前に消える**事故になる。

### 【知識 11】PC改行設計ルール — 文字数計算プロトコル

**なぜルール化するか：**
AIは「コードから1行に入る文字数を逆算」できる。やっていないだけ。
日本語は等幅に近いので、`font-size` と `コンテナ幅` から1行文字数を計算でき、これにより**ブラウザを見なくても泣き別れを予測できる**。
従来「実機で見れば分かる」と無意識に判断してきた箇所は、すべて計算に置き換える。

**文字幅の基本式：**

| 種別 | 1文字あたり幅 |
|---|---|
| 全角日本語（ひらがな・カタカナ・漢字） | `font-size × 1.0` |
| 半角英数記号 | `font-size × 0.5` |
| 半角カナ | `font-size × 0.5` |

**1行文字数の計算式：**

```
1行文字数 = floor((コンテナ実幅 − padding×2) / font-size) − 2
                                                          ↑ 安全マージン
```

**計算例：**
- `max-width: 760px / padding: 0 / font-size: 20px` → `floor(760/20) - 2 = 36字/行`
- `max-width: 640px / padding: 16px×2 / font-size: 18px` → `floor((640-32)/18) - 2 = 31字/行`
- `max-width: 480px / padding: 12px×2 / font-size: 16px` → `floor((480-24)/16) - 2 = 26字/行`

**泣き別れ判定（severity）：**

| レベル | 条件 | 対処 |
|---|---|---|
| critical | 最終行 ≤ 4字 または 全体の15%未満 | 即修正 |
| warning | 最終行 ≤ 8字 または 全体の25%未満 | 改善推奨 |
| ok | 最終行 > 8字 かつ 25%以上 | 許容 |

**`.br-pc` 挿入位置の優先順位：**

```
A. 意味の切れ目: 読点「、」 > 句点「。」 > 助詞「を/に/で/は/が」の後
B. 分断禁止:    カタカナ語・複合名詞・固有名詞の途中で切らない
C. バランス:    1行目 : 2行目 = 0.8〜1.0 : 1.0〜1.2（順三角形）
D. 孤立回避:    最終行が極端に短くならないこと

判断フロー:
  1. C（バランス）でだいたいの位置を決める
  2. その近辺で A（読点・句点）に合う点を探す
  3. B（分断禁止）に違反していないか確認
  4. D（孤立）が問題ないか確認
```

**分断禁止辞書（YAMLの `no_split_dictionary` を参照）：**
代表例：成果物 / 動画制作 / マンツーマン / ワークフロー / 個別HTML教材
これらが行末・行頭で分断されている場合は **critical 即修正**。

**保護方法（強制 nowrap）：**

```html
<span class="nb">マンツーマン伴走コース</span>
```

```css
.nb {
  display: inline-block;
  white-space: nowrap;
}
```

### 【知識 12】カードグリッド並列性ルール

**最優先原則：「同セクションの同種カードは行数を揃える」**

| 項目 | 優先度 | ルール |
|---|---|---|
| 見出し行数 | critical | 全カード1行揃え or 全カード2行揃え。1枚だけ違うのは最悪 |
| 説明文行数 | warning | 2〜3行で統一。**1行は説明不足感が出るので最低2行** |
| 下部ラベル位置 | warning | 本文行数を揃えれば自動で揃う |

**揃える方向：常に「上向き（盛り）」**

```
NG思考: 「短いカードに合わせて全部1行に削る」 → 情報が痩せる
OK思考: 「長いカードに合わせて全部2-3行に盛る」 → リッチになる
```

**文字を盛る時の品質基準：**

- OK: 副詞・接続詞で自然に長くする（事前に / 一気通貫で / 必ず）
- OK: 動詞を「です・ます」化（明確化 → 明確化します）
- OK: 起点・終点を明示（〜から、〜まで）
- OK: 対象・効果を明示（〜により、〜のため）
- NG: 意味のない水増し（とても / 本当に / しっかり連発）
- NG: 重複表現（明確に明確化する）
- NG: ニュアンスを変える言い換え

**短縮する場合の禁止事項（ニュアンス破壊）：**

- 硬い熟語化禁止（〜不可、〜不能など）
- 逆接ニュアンス削除禁止（「見ても」「ても」「のに」）
- お客様の声の勝手な短縮禁止

**修正の選択肢（優先順）：**

```
1. カード幅を調整する（広げる・絞る）
2. 文字数を盛る（長いカードに合わせる）
3. .br-pc を入れる（意味＋バランスで判断）
4. 文字数を削る（最終手段・ニュアンス維持必須）
5. フォントサイズ調整（最後の最後）
```

### 【知識 13】レイアウトスコープ判断フロー

**「改行だけ直すAIは無能」。上流から判断せよ。**

```
Level 1: カード単体
Level 2: カードグループ（同セクションの兄弟カード）
Level 3: 2カラムバランス（左右カラム間の重量感）
Level 4: セクション全体
Level 5: ページ全体のリズム
```

**判断の順序：上流（スコープ）から決める。下流（改行）から見ない。**

**2カラム特有の制約：**

```
2カラムは「反対カラムとgap」で横幅が自動決定される
  → 横幅自由度が低い
  → 文字数調整 or 改行調整がメイン手段
  → 改行ヘビーで右余白が増えるとレイアウト劣化

シーソー型解決:
  - 大カードを 10〜15% 縮小
  - その分を小カード群に分配
  - 必要なら小カード群のフォントサイズを1段階下げる
```

**右余白の許容範囲：**

| 状況 | 許容範囲 |
|---|---|
| 自然折り返し時 | 10%以内が理想 |
| `.br-pc` 強制改行時 | 20%以内まで許容 |
| それ以上 | 設計ミスと判定 |

**コンテンツ種別ごとの厳しさ：**

| 種別 | 厳しさ | デフォルト方向 | 理想行数 |
|---|---|---|---|
| ヒーロー大見出し | ★★★★★ | 短く明快 | 2行 |
| ヒーロー下リード文 | ★★★★★ | バランス重視 | 2-3行 |
| カードグリッド（特徴） | ★★★★ | 盛り（並列性） | 2-3行揃え |
| カードグリッド（ステップ） | ★★★★ | 盛り（並列性） | 1-2行揃え |
| お客様の声 | ★★★ | 盛り（リアリティ） | 2-3行揃え |
| FAQ | ★★ | 可読性優先 | 自由（横幅800-900pxに絞る） |
| **Final CTA** | **★★★★★** | **短く明快** | **2行（3行以上禁止）** |
| フッター・注記 | ★ | 情報伝達 | 自由 |

**最悪パターンの覚え方：**

```
「2行で上下バランスが悪い」 = 最悪
  → 2行は逃げ場がない。比率がモロに見える
「3行以上でバランスがそこそこ」 = 許容
  → 中の行がバッファになる
```

**1行ギリギリ問題：**

```
1行で右端まで使用率95%以上 = 警告
  → 視覚的に窮屈・呼吸がない
対処: 読点位置で意味のある折返しを入れる（上短下長で）
     または カード幅を広げる
```

### 【知識 14】SP（スマホ）設計ルール【v1.3 大改訂】

**前提：SPは見た目勝負**

```
PC: 端末幅がほぼ固定（1280/1440/1920）→ 厳密な文字数計算が当たる
SP: 端末幅が多種多様（360/375/390/414/430）→ 厳密計算は当たらないが、
                                              基準デバイスで判定する
```

**SPの大原則（ 流）：**

```
1. SPは「自然折り返し」と「.br-sp 強制改行」の二択で構築
2. CSSの自動バランス系（text-wrap: balance 等）は絶対使用禁止
   → 日本語アルゴリズムが効かず、カタカナ分断を起こす
3. 無印 <br> は絶対に書かない（critical違反）
   → PC意図がSPに引きずられ、自然折り返しと衝突して泣き別れを生む
4. PC用クラスとSP用クラスは必ず分ける
   → 同一クラスで両方制御するとPC調整がSPを壊す
```

#### 基準デバイス：**Samsung Galaxy S8+ / 360px**

```
SPの検証はこの幅で行う。これで問題なければ合格。
320px以下の極小端末は対象外と割り切る
（それに合わせると上の幅で必ず破綻する）
```

#### `.br-sp` 使用ポリシー（v1.0からの方針転換）

**従来（v1.0）：`.br-sp` 原則禁止**
**現行（v1.1〜）：状況判断で計算して使う**

`.br-sp` を使うべきケース：
- 自然折り返しで**複合語・カタカナ語・固有名詞が分断**される
- 自然折り返しで**最終行が1〜2文字だけ落ちる**
- **360pxで改行を入れても左右に余裕がある**

自然折り返しのままにすべきケース：
- 自然折り返しで2行のバランスが取れている
- 改行を入れると360pxで横はみ出しリスクがある
- 説明文など長文で改行制御が無意味な箇所

**判断原則：「1〜2文字孤立を放置するより、明示的改行で2行固定する方が美しい」**

例：
```
NG（自然折り返し）:
  動画制作フローの完成イメ
  ージ                    ← カタカナ「イメージ」分断

OK（.br-sp で固定）:
  動画制作フローの<br class="br-sp">完成イメージ
  → 360pxでも左右に余裕、分断回避
```

#### 絶対禁止項目（critical違反）

| 違反 | 理由 | 対処 |
|---|---|---|
| 無印 `<br>` | PC意図がSPに引きずられ事故 | `<br class="br-pc">` / `<br class="br-sp">` を明示 |
| `text-wrap: balance` | 日本語でカタカナ分断 | `text-wrap: pretty` |
| `word-break: keep-all` | 句読点が単独で取り残される | 指定しない |

#### SP用フォントサイズ・行間設計（必ず `clamp()`）

```css
h1 { font-size: clamp(28px, 7.5vw, 76px); line-height: 1.2; }
h2 { font-size: clamp(24px, 5vw, 58px); line-height: 1.25; }
.card-heading { font-size: clamp(18px, 4.5vw, 24px); line-height: 1.3; }
body { font-size: clamp(14px, 3.8vw, 16px); line-height: 1.7; }
```

`line-height` の指針：
- 大見出し（太字大サイズ）：**1.2〜1.35**
  - 狭すぎ → 圧迫感・読みにくい
  - 広すぎ → バラけて見える
- 本文：1.7〜1.8

#### PC/SP クラス分離パターン

```css
/* OK: 共通クラスをメディアクエリで切り分け */
.hero-sub { /* 共通スタイル */ }
@media (min-width: 641px) { .hero-sub { font-size: ... } }
@media (max-width: 640px) { .hero-sub { font-size: ... } }

/* NG: PC前提の改行・幅指定を直書き */
.hero-sub { /* PC用の幅・改行が SP まで効く */ }
```

#### よくあるSP事故と対処

| 事故 | 対処 |
|---|---|
| ボタン内テキストの折り返し（critical） | ボタンに `white-space: nowrap` or 幅を広げる |
| アイコンが大きすぎて本文の横幅が狭い | SP用メディアクエリでアイコン縮小（例: 52px→40px） |
| カード下に謎の余白 | `align-items: stretch` を SP では `start` に。`min-height` を SP では `unset` |
| 横スクロール | 既存 Layer 1/2 で対処済み |
| フォントが大きすぎて1行に2〜3字 | `clamp()` で SP を控えめに |
| 長い英単語・URLの overflow | 既存 `overflow-wrap: anywhere` で対処済み |
| カタカナ・複合語の分断 | `<span class="nb">` で nowrap 保護 + 必要なら `.br-sp` |

#### SPでの許容範囲

- 説明文が4〜5行になっても許容
- カード高さバラつきはPCほど気にしない（縦並びだから）
- 右余白判定は緩める

## Phase 2.5：日本語コピー記述プロトコル（必須実行）

**日本語のコピーを書く・直す場面では、毎回必ずこのプロトコルを実行せよ。**
**目視で「だいたい大丈夫だろう」は禁止。計算なしでコピーを書くのも禁止。**

### 実行手順

```
Step 1: 対象要素のCSSを特定
  → max-width / padding / font-size を読む（または親要素から推定）

Step 2: 1行文字数を計算
  → floor((max-width − padding×2) / font-size) − 2

Step 3: コピー候補の総文字数をカウント

Step 4: 行数と最終行文字数を算出
  → 行数 = ceil(総文字数 / 1行文字数)
  → 最終行文字数 = 総文字数 − (行数−1) × 1行文字数

Step 5: severity判定
  - 最終行 ≤ 4字 or < 15% → critical
  - 最終行 ≤ 8字 or < 25% → warning
  - それ以外 → ok

Step 6: 分断禁止辞書とのマッチング
  → 行末・行頭付近で「成果物」「マンツーマン」等が割れていないか確認

Step 7: critical / warning なら修正案を提示
  → 修正の選択肢を優先順位通りに試す
  → 最終的に [計算表 + 修正後のコピー] を出力してから実装

Step 8: カードグリッドの場合は並列性確認
  → 兄弟カードの行数と揃っているか
  → 揃っていなければ「盛り方向」で調整

Step 9: SP（360px）での自然折り返しを脳内シミュレーション
  → 自然折り返しで「1〜2文字孤立」or「複合語分断」が起きるか
  → 起きるなら .br-sp で強制改行（読点・意味の切れ目で）
  → 360pxで改行後も左右に余裕があることを確認

Step 10: 禁止HTMLが混入していないか確認
  → 無印 <br> が残っていないか（critical）
  → 全ての <br> に .br-pc / .br-sp クラスが付いているか
```

### 必須出力フォーマット

コピーを書く・直す前に、以下の計算表を**必ず出力**してから実装に進む：

```
[文字数計算表]
要素: .hero-sub
コンテナ: max-width 760px / padding 0 / font-size 20px
1行文字数: 36字（PC基準）

コピー: 「企画・台本・素材生成・編集・記事化・投稿準備まで、あなた専用の動画制作ワークフローとして整えるマンツーマン伴走コースです。」
総文字数: 62字

[PC評価]
自然折り返し: 2行
最終行: 6字 ❌ critical
分断リスク: 「マンツーマン」がカタカナ分断

[SP評価 / Galaxy S8+ 360px]
自然折り返し時の予測: 「マンツーマン」分断の可能性 ❌
.br-sp で読点位置に強制改行を入れて分断回避

[修正後 HTML]
  企画・台本・素材生成・編集・記事化・投稿準備まで、<br class="br-pc">
  あなた専用の動画制作ワークフローとして整える<span class="nb">マンツーマン伴走コース</span>です。

結果:
  PC 1行目 28字 / 2行目 34字 ✅ 順三角バランス
  SP 360px: 「マンツーマン伴走コース」を .nb で保護、自然折り返しで分断回避 ✅
```

### 禁止事項（critical 違反）

- **無印 `<br>` を1つでも書く**（必ず .br-pc または .br-sp を付ける）
- **`text-wrap: balance` を使う**（日本語でカタカナ分断を起こす）
- **`word-break: keep-all` を使う**（句読点が単独で取り残される）
- **PC用クラスとSP用クラスを混在させる**（PC調整がSPを壊す）
- 計算表なしでコピーを書く
- 目視で「だいたい大丈夫だろう」と判断する
- PC側だけ計算してSP側（Galaxy S8+ 360px）を無視する
- 分断禁止辞書を確認せずに改行する
- カードグリッドで並列性を確認せずに個別最適化する
